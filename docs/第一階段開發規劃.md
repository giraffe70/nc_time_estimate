## 第一階段時間估測方法

第一階段建議使用「幾何時間 + 事件時間 + 簡化機台限制」。

對 G01/G02/G03：

```
time = path_length / feedrate
```

其中 feedrate 單位要統一成 mm/s。

對 G00：

```
time = path_length / rapid_feed
```

但更嚴謹一點，G00 不是單純沿直線等速移動。不同控制器可能是各軸獨立快速移動，因此可以用各軸時間取最大值：

```
time_x = abs(dx) / rapid_x
time_y = abs(dy) / rapid_y
time_z = abs(dz) / rapid_z

G00_time = max(time_x, time_y, time_z)
```

這比單純用三維距離除以快速進給更接近實際機台。

對 M06 換刀：

```
time += tool_change_sec
```

對 M03 主軸啟動：

```
time += spindle_start_sec
```

對 G04：

```
time += dwell_time
```

第一階段估測結果雖然還不是高精度數位雙生，但已經比單純讀 CAM 預估時間更透明，因為它能拆出每段時間來源。

## NC-Code 解析實作重點

NC-Code 解析要注意幾個坑。

第一，G-code 有模態延續。例如：

```
G01 X10 F500
Y20
X30
```

後兩行仍然是 G01，不能當成未知指令。

第二，座標可能只更新部分軸。例如：

```
G01 X10 Y20 Z-5
X20
```

第二行只有 X 改變，Y 與 Z 沿用前一個位置。

第三，圓弧要根據平面判斷。G17 是 XY 平面，G18 是 XZ 平面，G19 是 YZ 平面。G02/G03 可能使用 IJK，也可能使用 R。第一階段建議先完整支援 IJK，R 圓弧後續再補強，因為 R 有大小圓弧歧義。

第四，G95 進給模式要用主軸轉速換算：

```
G95 模式下：
實際進給 mm/min = F(mm/rev) × S(rpm)
```

如果程式有 G95 但沒有 S，系統必須提示「無法換算實際進給」。

第五，G04 dwell 要計算停留時間，但不同控制器的 P、X 單位可能不同，需要放到 controller profile 裡設定。

```

Y100

```

```
三行都應該視為 G01
F 都應該延續為 1000
```

## 測試 3：相對座標

```

G01 X10 F600
X10
X10
```

```
最終 X = 30
總長度 = 30 mm
```

## 測試 4：G95 進給

```
G21 G90
S1000 M03
G95
G01 X100 F0.2
```

預期：

```
實際進給 = 0.2 mm/rev × 1000 rev/min = 200 mm/min
```

---

## 測試 5：G00 軸向同動

```
G00 X100 Y50 Z10
```

如果：

```
X rapid = 10000 mm/min
Y rapid = 10000 mm/min
Z rapid = 5000 mm/min
```

則：

```
tx = 100 / (10000/60)
ty = 50 / (10000/60)
tz = 10 / (5000/60)

G00 time = max(tx, ty, tz)
```

# 第一階段最重要的開發原則

第一階段不要追求一開始就完全模擬控制器，而是要把「資料結構與流程」設計正確。

最核心的設計原則是：

```
1. Parser 不直接算時間。
2. Parser 只負責把文字轉成結構化資料。
3. Modal State 負責還原 CNC 當下狀態。
4. IR 負責保存加工語意。
5. Geometry Processor 負責算路徑。
6. Time Estimator 負責算時間。
7. S 曲線、Look-ahead、五軸運動學先保留接口。
```

用一句話說，第一階段要先做出「乾淨的 NC-Code 執行紀錄」。只要這個 IR 建得好，第二階段要加入 S 曲線、前瞻控制、機台校正，就只是替換 Motion Simulation Engine，而不是重寫整套系統。
